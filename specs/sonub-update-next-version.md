## 1. 채팅 기능 데이터베이스 전략
- **배경**: 채팅 기능을 전부 Firestore에 저장할 경우 메시지마다 `/chat-joins`, `/users/{uid}/newMessageCount` 등을 갱신하면서 읽기/쓰기 비용이 폭증할 수 있음.
- **트래픽 가정**: 그룹 채팅방 100개 × 방당 1,000명 × 일 100건 메시지 × 파생 업데이트 2회 × 30일 ≈ 6억 회 문서 쓰기 + Cloud Functions 문서 읽기 → Firestore 비용 부담이 큰 구조.

### 1.1 Firestore vs RTDB 비용 비교 (요약)
| 항목 | Firestore | RTDB | 메모 |
|------|-----------|------|------|
| 저장 공간 1GB | 약 \$0.18 | \$5 | Firestore가 20배 이상 저렴 |
| 다운로드 1GB | 약 \$0.12 | \$1 | Firestore가 8배 이상 저렴 |
| 읽기/쓰기 | 과금 O | 과금 X | RTDB는 트랜잭션 비용이 없음 |

> 결론: 저장/다운로드는 Firestore가 거의 공짜에 가깝지만, 초당 수많은 문서 read/write가 발생하면 Firestore가 오히려 비싸질 수 있음.

### 1.2 현재 전략: RTDB 완전 사용

**결정 사항**: 채팅 기능과 회원 데이터는 **완전히 RTDB에서 처리**합니다.

#### 1.2.1 RTDB 사용 이유: Fan-out 문제

**그룹/오픈 채팅방의 Fan-out 특성**:
- 1개의 메시지가 발생하면 → 채팅방 참여자 수만큼 `/chat-joins/{uid}/{roomId}` 업데이트 발생
- 예시: 1,000명 참여 채팅방에서 메시지 1개 → 1,000개의 `/chat-joins` 노드 업데이트 필요
- 하루 100건 메시지 × 1,000명 = **100,000회 쓰기 작업** (단일 채팅방)
- 채팅방 100개 × 일 100건 × 1,000명 = **일 1,000만 회 쓰기**

**Firestore vs RTDB 비용 영향**:

| 시나리오 | Firestore 비용 | RTDB 비용 |
|---------|--------------|----------|
| 일 1,000만 회 쓰기 | 약 \$540/월 (쓰기 건수 과금) | 포함 (트래픽 과금만) |
| 월 3억 회 쓰기 | 약 \$16,200/월 | 데이터 크기에 따라 \$0~수백 달러 |

**결론**:
- Fan-out이 발생하는 채팅 시스템에서는 **RTDB가 압도적으로 유리**
- Firestore는 건수당 과금이므로 fan-out 환경에서 비용 폭탄 위험
- RTDB는 트래픽(바이트) 과금이므로 작은 데이터 여러 번 쓰기에 유리

#### 1.2.2 현재 RTDB 구조

**채팅 관련 노드**:
- `/chat-rooms/{roomId}`: 채팅방 메타데이터 (이름, 설명, owner 등)
- `/chat-messages/{roomId}/{messageId}`: 메시지 본문
- `/chat-joins/{uid}/{roomId}`: 사용자별 참여 채팅방 목록 (fan-out 발생)
- `/chat-invitations/{uid}/{roomId}`: 채팅 초대 목록
- `/chat-room-passwords/{roomId}`: 채팅방 비밀번호

**회원 관련 노드**:
- `/users/{uid}`: 사용자 프로필 및 메타데이터

### 1.3 차기 버전 마이그레이션 계획

**비용 발생 시 단계적 Firestore 이전 전략**:

1. **Fan-out이 발생하지 않는 노드만 Firestore로 이전**:
   - ✅ `/chat-rooms/{roomId}` → Firestore로 이전 가능
   - ✅ `/chat-messages/{roomId}/{messageId}` → Firestore로 이전 가능
   - ❌ `/chat-joins/{uid}/{roomId}` → **RTDB에 유지** (fan-out 발생)
   - ✅ `/users/{uid}` → Firestore로 이전 가능

2. **마이그레이션 시 이점**:
   - `chat-rooms`: Firestore의 강력한 쿼리/필터 활용 (예: 카테고리별 검색, 태그 검색)
   - `chat-messages`: Firestore의 낮은 저장 비용 활용 (메시지 누적 시)
   - `users`: Firestore의 복잡한 쿼리 활용 (사용자 검색, 필터링)

3. **RTDB 유지 노드**:
   - `/chat-joins/{uid}/{roomId}`: Fan-out 발생하므로 **영구적으로 RTDB 유지**
   - `/chat-invitations/{uid}/{roomId}`: 실시간성 + fan-out 때문에 RTDB 유지

4. **하이브리드 운영 시 고려사항**:
   - Cloud Functions에서 Firestore와 RTDB 동시 업데이트 필요
   - 동기화 일관성 보장 (트랜잭션 또는 보상 처리)
   - 장애 대응 플로우 정의 (한쪽 실패 시 롤백 또는 재시도)

**마이그레이션 시기**:
- RTDB 트래픽 비용이 월 \$500 이상 발생 시 검토
- Firestore 쿼리 기능이 반드시 필요한 경우 (예: 고급 검색 기능)
- 메시지 보관 용량이 10GB 이상 누적되어 저장 비용이 부담될 때

## 2. Firestore vs RTDB 상세 비용 비교

| 항목 | Firestore | Realtime Database | 비고 |
|------|-----------|-------------------|------|
| 저장(GB/월) | 약 \$0.18 (1GB 무료) | \$5 (1GB 무료) | Firestore가 약 20~30배 저렴 |
| 다운로드(GB) | \$0.12 (10GB 무료) → 대량 구간 \$0.11 | \$1 (10GB 무료) | 2TB 전송 시 Firestore ≈\$230, RTDB ≈\$2,000 |
| 읽기/쓰기 | 100k당 \$0.06/0.18, 삭제 \$0.02 | 별도 과금 없음 (트래픽에 포함) | 하루 50k 읽기/20k 쓰기 무료 |
| 동시 연결 | 최대 1M 연결 지원, 리스너 유지 비용 없음 | 인스턴스당 200k 연결, 유지 비용 없음 | 변경 시 Firestore는 읽기 건수, RTDB는 전송 바이트가 과금 |

### 2.1 저장 비용
- Firestore: 5GB 저장 시 무료 1GB 제외 약 \$0.72/월. 사실상 미미.
- RTDB: 5GB 저장 시 무료 1GB 제외 약 \$20/월. 대용량일수록 부담이 큼.

### 2.2 읽기/쓰기 요청
- Firestore: 100만 읽기/일 → 월 3천만 건 → 무료분 제외 약 \$17.1/월. 쓰기 10만/일 → 월 300만 건 → 무료분 제외 약 \$4.32/월. 합계 약 \$20 안팎.
- RTDB: 요청당 비용은 없지만 전송량으로 과금되므로 데이터가 커질수록 다운로드 비용이 누적됨. 문서 크기가 200B 미만일 때만 Firestore보다 유리하다는 사례가 있음.

### 2.3 다운로드(아웃바운드)
- Firestore: 2TB 전송 시 첫 10GB 무료, 이후 구간별 \$0.12~0.11/GB → 약 \$230~240/월.
- RTDB: 2TB 전송 시 (10GB 무료 제외) 2038GB × \$1 = 약 \$2038/월.

### 2.4 동시 연결 및 리스너
- Firestore: 1,000명 수준은 부담 없음. 변경 시 해당 문서가 1,000명에게 전파되면 1,000건의 읽기로 계산.
- RTDB: 동일 변경이 발생하면 데이터 크기 × 접속자 수만큼 트래픽이 발생. GB당 \$1 요율 때문에 업데이트 빈도가 높으면 비용 급증.

### 2.5 무료 할당량
- Firestore/RTDB 모두 1GB 저장, 10GB 전송, 일부 읽기/쓰기 무료지만 본 시나리오 규모(5GB 데이터, 2TB 전송, 일 100만 읽기)에서는 영향 미미.

### 2.6 최종 결론

**현재 전략 (RTDB 완전 사용)**:
- ✅ **Fan-out 환경에서 RTDB가 압도적으로 유리**
  - Firestore: 건수당 과금 → fan-out 시 비용 폭발
  - RTDB: 트래픽 과금 → 작은 데이터 여러 번 쓰기에 최적
- ✅ **그룹/오픈 채팅의 특성상 RTDB 필수**
  - 1개 메시지 → 수백~수천 개 업데이트 발생
  - Firestore 사용 시 월 \$16,200 → RTDB 사용 시 수백 달러 이하
- ✅ **실시간성과 동시 연결**
  - RTDB는 200k 동시 연결 지원, 실시간 동기화 최적화

**차기 버전 마이그레이션 전략**:
- ⏳ **조건부 Firestore 이전** (RTDB 비용이 월 \$500 초과 시)
  - Fan-out 없는 노드만 이전: `chat-rooms`, `chat-messages`, `users`
  - Fan-out 노드는 영구 RTDB 유지: `chat-joins`, `chat-invitations`
- ⏳ **Firestore 이점 활용**
  - 저장 비용 절감 (GB당 \$0.18 vs \$5)
  - 강력한 쿼리/필터 기능 (검색, 카테고리별 조회)
  - 낮은 다운로드 비용 (GB당 \$0.12 vs \$1)

**핵심 원칙**:
- 🔥 **Fan-out 발생 노드 = 영구 RTDB**
- 💾 **Fan-out 없는 노드 = 조건부 Firestore 이전 가능**
- 💰 **비용 발생 전까지는 RTDB 완전 사용**
